Order-mode reorder interaction lab

Card chrondle-ux-order-interaction. Live evidence (2026-07-06, /order #239): the drag handle is a row of four decorative dots (~30px tall, below the 44px floor), it has no accessible name, and there is no visible keyboard path — OrderEventList.tsx wires a dnd-kit KeyboardSensor but nothing on screen tells a keyboard or screen-reader user it exists. The event-text area is simultaneously a "tap to read more" target and, in several of these options, adjacent to the drag surface. 8 structurally distinct interaction patterns (R1–R8). Constraint: reordering must work by touch-drag, by a discrete/step control, and by keyboard, per the card's stated outcome; drag-initiation and text-expand must be non-overlapping targets ≥44px.

Structure

  • DotsSixVertical grip icon, 44×44px, aria-label="Reorder [event], position N of M"; drag still works for pointer/touch via the existing dnd-kit sensors.
  • Two 44×44px <button>s (Move up / Move down), disabled at the list boundaries, wired to the same onOrderingChange path drag already uses.
  • Keyboard: buttons are plain focusable buttons — Tab + Enter/Space, no gesture to learn.
  • Custom accessibility.announcements on DndContext covers drag pickup/move/drop/cancel; a second aria-live="polite" region covers stepper-triggered moves.

For

  • Directly matches the card body's own worked example ("tap-to-select then move, or up/down"). Three non-overlapping targets (handle, text, steppers), each independently ≥44px. Native buttons are trivially testable and reliably announced by every screen reader without relying on custom drag semantics. Touch-drag is preserved for players who already use it — zero regression.

Against

  • Adds a third column to the card, narrowing the text column slightly on small screens (mitigated: badge/controls are fixed-width, text still gets the majority of the row).

Structure

  • Tap/Enter a card to select it (visibly highlighted, announced). Tap/Enter a second card to place the first one immediately before it. No drag at all.

For

  • Fully keyboard/touch/SR-native (two button presses); no drag gesture required anywhere.

Against

  • Replaces drag entirely — regresses the many players who already reorder fine by dragging. Two-tap grammar for a 3-6 item list is a bigger interaction change than the card's "outcome, not redesign" framing calls for, and needs its own placement-target affordance (which card do you tap to mean "move to end"?).

Structure

  • A toolbar toggle switches the list into an explicit edit mode; each row gets a native <select> assigning its position 1–N. Toggling off returns to a read-only, drag-free view.

For

  • Native <select> is maximally accessible by construction (every platform's picker UI is already a11y-audited).

Against

  • Introduces a whole second mode/toolbar for what is normally a 3-6 item list — heavier surface area than the card's "bars, not a design" framing wants, and reassigning positions via a dropdown while N other rows silently shift is a confusing model for a guess-the-order puzzle (you're not looking at a settings form).

Verdict (delegated mode — operator may re-verdict)

R2 — drag handle (enlarged, labeled) + always-visible Move Up/Down steppers. It is the only option built from three simultaneously-satisfied, independently ≥44px targets: the redesigned grip handle (pointer/touch drag, unchanged mechanism, zero regression for existing players), the event-text button (tap-to-expand, untouched), and a new stepper column (the card's own named example of a "discrete control"). Steppers are plain <button> elements, so the keyboard and screen-reader bars are met by construction — no custom gesture to teach or discover — while drag stays available via the same dnd-kit sensors already wired in OrderEventList.tsx, now paired with content-aware accessibility.announcements for pickup/move/drop/cancel.

R1 and R8 fail the discrete-control bar outright (R1 only relabels the same undiscoverable keyboard gesture; R8 explains it once and then hides it again). R5 fixes touch ergonomics but not keyboard/SR discoverability, so it would still need R2's steppers layered on — it's a subset, not an alternative. R3/R4/R6 all replace drag with a different primary grammar, which is a bigger interaction rewrite than "bars, not a design" calls for, and each trades away either drag continuity (R3/R4/R6) or per-move speed (R4, R7). R7 is the closest runner-up — keeping it in mind if steppers ever feel cramped on very narrow viewports — but costs an extra tap per move versus R2's direct buttons.
